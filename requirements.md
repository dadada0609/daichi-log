# requirements.md

## 1. プロジェクト概要 (Project Overview)
本プロジェクトは、Backlogの主要機能である「課題管理」「ガントチャート」「Wiki」「Git連携」を統合したプロジェクト管理ツールのコア機能を構築する。

## 2. コア機能要件とデータ依存関係 (Core Features & Dependencies)

### 2.1 課題 (Issues)
* **概要**: プロジェクトの最小管理単位であり、システムの中核エンティティ。
* **データ構造**:
  * `IssueKey` (例: PROJ-123)
  * `Title` (件名)
  * `Description` (詳細 - Markdown対応)
  * `Status` (未対応, 処理中, 処理済み, 完了)
  * `Priority` (低, 中, 高)
  * `Assignee` (担当者)
  * `StartDate` (開始日)
  * `DueDate` (期限日)
  * `EstimatedHours` (予定時間)
  * `ActualHours` (実績時間)
* **他機能との依存**: 本システムの中心エンティティであり、Gantt, Wiki, Gitの全機能の紐付け元となる。

### 2.2 ガントチャート (Gantt Chart)
* **概要**: 課題のスケジュールをタイムライン上に視覚化するビュー層。
* **依存関係 (課題 -> ガントチャート: 1対1)**:
  * 課題の `StartDate` と `DueDate` をX軸の描画パラメータ（期間）として使用する。
  * 課題の `Status` を進捗ステータスとして色分け等で表現する。
* **制約**: `StartDate` と `DueDate` を持つ課題のみが期間バーとして描画される。片方のみの場合はマイルストーンまたは点として描画される。

### 2.3 Wiki
* **概要**: プロジェクトごとのドキュメント管理機能。
* **依存関係 (課題 <-> Wiki: 多対多)**:
  * **相互リンク**: Wiki本文内に `IssueKey` を記述した場合、課題ページへのハイパーリンクへ自動変換する。
  * 課題の詳細やコメントからWikiページへのリンク参照も同様に処理される。
* **データ構造**:
  * `PageID`, `Title`, `Content` (Markdown), `Version`, `CreatedBy`, `UpdatedBy`

### 2.4 Git
* **概要**: リポジトリのコミット履歴と課題を統合する連携機能。
* **依存関係 (課題 <-> Git: 多対多)**:
  * **コミット連携**: コミットメッセージ内の `IssueKey` (例: `#PROJ-123`) をパースし、該当する課題の「関連コミット」または「コメント」として履歴を自動追記する。
  * **ステータス自動遷移**: 特定のキーワード (例: `fix #PROJ-123`) が含まれる場合、該当課題の `Status` を「処理済み」に自動更新する。
* **データ構造**:
  * `RepositoryID`, `CommitHash`, `Author`, `Message`, `Timestamp`

## 3. 依存関係の定量的分解
各機能モジュール間の関係性を定量的に示す。
* **Gantt Chart表示の計算量**: 1プロジェクトあたりアクティブな課題数を `N` とした場合、ガントチャート描画のためのデータフェッチは `O(N)`。
* **Git連携の処理量**: 1回のPushに含まれるコミット数を `C` とした場合、メッセージパース（正規表現マッチング）による課題特定処理の負荷は `O(C)`。1コミットが複数の課題キーを含む場合があるため、更新クエリの最大発生数は `C × 課題キー数` となる。
* **Wikiのパース量**: ページ文字数を `W` とした場合、表示時のハイパーリンク置換処理（正規表現）の計算量は `O(W)`。

## 4. 内部検証プロセス (CoVe: Chain of Verification) 結果

### <VerificationQuestions>
1. **事実確認**: Backlogの公式仕様において、ガントチャートは「課題」のどのフィールドに依存して描画されるか？
2. **仕様の整合性**: Gitのコミットメッセージによる課題連携は、システム的にどのような依存関係（1対N、多対多など）を生むか？
3. **論理的実現性**: 要求された4つの機能は互いに論理的な矛盾を生まずに単一のデータベーススキーマで表現可能か？

### <Verify>
1. **事実確認の検証**: Backlogの仕様上、ガントチャートのバー描画には課題の「開始日 (Start Date)」と「期限日 (Due Date)」が必須パラメータとなる。これらを持たないタスクは期間として描画できないため、システム的に制限やフォールバック処理を考慮する必要がある。要件定義「2.2 ガントチャート」にこの制約を反映した。
2. **仕様の整合性の検証**: 1つのコミットが複数の課題キーを含むこと（例: `fix #PROJ-1, #PROJ-2`）が可能であり、また1つの課題に対して複数のコミットが紐づく。したがって、Gitコミットと課題のデータ的依存関係は「多対多 (M:N)」である。要件定義「2.4 Git」にこれを反映した。
3. **論理的実現性の検証**: 課題(Issue)をコアルートエンティティとし、Wikiは独立したエンティティ（テキストパースによりIssueを動的参照）、Git連携はWebhook等を介したイベント駆動のIssue更新処理としてモデリング可能である。機能間に循環参照や排他制御の論理的矛盾は見られず、RDB等の一般的なアーキテクチャで実装可能である。
